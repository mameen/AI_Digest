# Always-On Agents: A Survey of Persistent Memory, State, and Governance in LLM Agents

## 1. Overview

Always-on agents are **persistent-state systems** whose behavior depends on durable state accumulated across interactions, beyond a single prompt or task. Unlike episodic LLM agents that reset after each task, always-on agents retain:

- **Memories** (episodic, semantic)
- **Preferences**
- **Procedural skills**
- **Task ledgers and commitments**
- **Permissions and credentials**
- **Provenance and audit trails**
- **Shared/social state**
- **Trigger state and external commitments**

> “An always-on agent is one whose behavior at a given moment depends on state accumulated before that moment, beyond the current prompt or task instance.”

The paper’s core claim: the field focuses heavily on **accumulating and retrieving state**, but far less on **governing, repairing, and relinquishing** it. To address this, the authors introduce the **Always-On Evaluation Protocol (AOEP-v0)**, which scores **state mutation and recovery obligations**, not just answer quality.

## 2. Six Diagnostic Axes of Persistent State

Each persistent-state item is characterized along six axes:

1. **Authority**  
   What permits this state to influence an action?  
   - Example: a stored preference silently changes default tool arguments.

2. **Scope**  
   Which user, task, tool, time, or group may use this state?  
   - Example: private memory leaking across users is a scope failure.

3. **Mutability**  
   Can the state be revised, superseded, decayed, or locked?  
   - Example: a locked policy vs. a mutable preference.

4. **Provenance**  
   Which source, timestamp, and transformation produced it?  
   - Example: a summary written months ago overriding fresh evidence.

5. **Recoverability**  
   Can derived state and affected decisions be rolled back?  
   - Example: undoing a poisoned workflow or misfired commitment.

6. **Actionability**  
   Is the state evidence, preference, policy, skill, or executable commitment?  
   - Example: a learned workflow that can be replayed as an action.

> “A memory that scores perfectly on recall can still be unsafe if it has no authority boundary, no provenance, and no path to repair after a bad action.”

## 3. Persistent-State Taxonomy

### 3.1 Classical Memory Types

- **Episodic traces**  
  Concrete records of past interactions or events.

- **Semantic facts**  
  Stable knowledge about the world or system.

- **Preferences**  
  User-specific choices, styles, and tendencies.

- **Procedural skills**  
  Learned workflows, scripts, or strategies that can be reused.

### 3.2 Control-Relevant State (Beyond Memory)

- **Workflow and task-ledger state**  
  Open commitments, obligations, and task progress.

- **Policy and permission state**  
  Access control, consent, and allowed operations.

- **Provenance and audit state**  
  Lineage of decisions, tool calls, and state changes.

- **Shared and social memory**  
  State shared across agents or users.

- **Trigger state**  
  Conditions under which the agent acts proactively.

- **Tool and credential state**  
  API keys, tokens, and tool configurations.

- **External commitments and side effects**  
  Calendar events, purchases, partial actions, deletions in progress.

> “External commitments are not internal memory records; they are part of the agent’s persistent accountability surface because they must remain traceable to the internal state that authorized them.”

## 4. The Persistent-State Lifecycle

The paper reframes memory as one stage in a broader lifecycle:

```mermaid
flowchart LR
    A[Observe] --> B[Write]
    B --> C[Validate]
    C --> D[Organize]
    D --> E[Retrieve]
    E --> F[Act]
    F --> G[Update]
    G --> H[Forget]
    H --> I[Audit]
    I --> J[Rollback]
```
